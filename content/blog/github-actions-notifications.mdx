## How I Fixed GitHub Actions Notifications (The Easy Way)

After that disastrous Tuesday, I spent exactly 15 minutes setting up proper GitHub Actions notifications with Echobell. Here's the exact setup that's been saving my team's sanity ever since:

### Step 1: Create Your "Build Failures" Channel

1. **Open Echobell** and create a new channel
2. **Name it something obvious** like "ðŸš¨ CI/CD Failures" (the emoji helps for quick recognition)
3. **Choose red** as the color (because failures deserve urgency)

```
Body: Repo: {{repository}} | Run: {{run_number}} | Check: {{run_url}}
```

5. **Choose "Time Sensitive"** notifications (you want to know, but maybe not at 3 AM unless it's critical)

### Step 2: Add the Webhook to Your Workflow (Copy-Paste Ready)

The [Echobell GitHub Action](https://github.com/weelone/echobell-action) makes this ridiculously simple. Here's the exact YAML I use for single-job workflows:

```yaml
name: Build and Test

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build
        run: npm run build

      - name: Test
        run: npm test

      - name: ðŸš¨ Notify on Failure
        if: failure()
        uses: weelone/echobell-action@v1
        with:
          webhook-url: ${{ secrets.ECHOBELL_WEBHOOK_URL }}
```

**Pro tip:** Add that webhook URL to your repository secrets as `ECHOBELL_WEBHOOK_URL`. You can copy it from your Echobell channel settings.

### Step 3: Handle Complex Multi-Job Workflows

For larger pipelines, you want to know exactly which part failed. Here's my pattern for multi-job workflows:

```yaml
name: Full Deploy Pipeline

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Tests
        run: npm test

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build Application  
        run: npm run build

  deploy:
    needs: [test, build]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to Production
        run: npm run deploy:prod

  # ðŸŽ¯ This is the magic - one job that catches all failures
  notify-failure:
    runs-on: ubuntu-latest
    needs: [test, build, deploy]
    if: always() && contains(needs.*.result, 'failure')
    steps:
      - name: ðŸš¨ Alert the Team
        uses: weelone/echobell-action@v1
        with:
          webhook-url: ${{ secrets.ECHOBELL_WEBHOOK_URL }}
```

This setup sends one clear notification whenever ANY part of your pipeline fails, with all the context you need to investigate.

## Pro Tips from 6 Months of Use

### Template Optimization
After experimenting with different formats, this is our winning notification template:

```
Title: ðŸš¨ {{repository}} build failed ({{branch}})
Body: Step: {{job_name}} | Run #{{run_number}} | Time: {{created_at}}
Link: {{run_url}}
```

This gives us everything we need: what failed, where, when, and how to investigate.

### Notification Urgency Strategy
- **Main/master branch failures**: Time Sensitive (breaks through Do Not Disturb during work hours)
- **Develop branch failures**: Normal notifications  
- **Feature branch failures**: We actually don't notify for these (developers see them in GitHub anyway)

### Team Sharing Workflow
1. **Create the channel** with good templates
2. **Share the subscription link** in your team Slack
3. **Let everyone choose their own urgency level** (this is key - don't force everyone to get called at 3 AM)
