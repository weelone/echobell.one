# Make.com + Echobell：あらゆる自動化をリアルタイムアラートに変える

Make.com（旧Integromat）は、使えば使うほどもっと使いたくなるツールです。Makeのライブラリには1,800以上のアプリがあり、手動で構築するなら何日もかかるような連携を実現できます。

Makeが単体でうまく解決できないこと——それは、自動化が実行されたとき、その結果がどうやって自分のもとに届くか、という問題です。

メールは遅く感じる。Slackの通知は積み重なって無視されがち。そして普通のプッシュ通知は……スマホが裏返しになっていたり集中モードだったりすると、通知が来ていないのと変わりません。

ここでEchobellが役立ちます。MakeのシナリオをEchobellのチャンネルに接続すれば、すべての選択肢が使えます。標準のプッシュ通知、集中モードを突き破る緊急アラート、そして待てない状況のための実際の電話呼び出し。

セットアップ方法と、実際に作る価値のある具体的なシナリオを紹介します。

## セットアップ（約3分）

**ステップ1：Echobellのチャンネルを作成する**

EchobellアプリでチャンネルをひとつつくりI、何を受け取るかがわかる名前をつけます——`Shopify緊急注文`、`Stripe決済エラー`など。Webhook URLをコピーします。

**ステップ2：MakeにHTTPモジュールを追加する**

Makeのシナリオで新しいモジュールを追加します。**HTTP > リクエストを実行**を検索して選択します。次のように設定します：

- **URL**：EchobellのWebhook URL
- **メソッド**：POST
- **ボディタイプ**：Raw
- **コンテンツタイプ**：application/json

**ステップ3：ペイロードを設定する**

ボディはこのような形式にします：

```json
{
  "title": "アラートのタイトル",
  "body": "発生した内容の詳細",
  "notificationType": "time-sensitive"
}
```

Makeのダイナミック値を使えば、シナリオの前のステップのデータを引き込めます。通知が実際に役立つ情報を伝えるようにしましょう。

これが基本パターンです。あとは適切なトリガーと緊急度を選ぶだけです。

## 作る価値のある5つのシナリオ

### 1. 高額Eコマース注文の通知

**トリガー**：Shopify / WooCommerce / Wix Stores — 一定金額以上の新規注文、または手動レビューが必要な注文

オンラインショップを運営していると、ダッシュボードを一日中眺めているわけにはいきません。でも、大きな注文が入ったとき、高リスク注文がレビューを要するとき、フルフィルメントの問題が発生したときは、すぐに知りたいはずです。

```json
{
  "title": "新規注文: {{order.name}}",
  "body": "{{customer.first_name}} {{customer.last_name}} — {{order.total_price}} {{order.currency}}",
  "notificationType": "time-sensitive"
}
```

不正フラグが立った注文には`calling`を使って、実際に電話が鳴るようにしましょう。

### 2. 新規リード通知

**トリガー**：Typeform、HubSpot、Salesforce、Pipedrive — 新規コンタクト作成、またはディールがホットなステージに移行

ファウンダー主導の営業や小規模な営業チームを持つ人にとって、新規リードへの素早い対応が成否を分けることがあります。リードがデモ申し込みをした瞬間にスマホが震えれば、相手がまだ自社のことを考えているうちにフォローアップできます。

```json
{
  "title": "新規デモリクエスト: {{contact.name}}",
  "body": "{{contact.email}} — {{form.company}} — 登録日時 {{now}}",
  "notificationType": "time-sensitive"
}
```

### 3. 決済エラーの通知

**トリガー**：Stripe、Paddle、Chargebee — サブスクリプション決済の失敗、チャージバックの発生

決済エラーは小さく見えて積み重なるものです。決済プロセッサーを監視するMakeシナリオがあれば、Echobellへの即時通知で素早く対応できます——顧客が解約したり請求に異議を申し立てる前に。

```json
{
  "title": "決済エラー: {{customer.email}}",
  "body": "{{invoice.amount_due}} {{invoice.currency}} — {{failure_message}}",
  "notificationType": "calling"
}
```

これは`calling`を使う価値があります——売上損失は今やっていることを中断する十分な理由です。

### 4. 在庫・業務アラート

**トリガー**：Airtable、Google Sheets、Notion — 行の変更、フィールドがしきい値を超える、ステータスの切り替え

多くの業務はスプレッドシートやデータベースで動いています。Makeでそれらを監視し、注意が必要なときに即座に知らせることができます。在庫が最低水準を下回る、配送ステータスが「失敗」に変わる、予定タスクが期限切れになるなど。

```json
{
  "title": "在庫不足: {{item.name}}",
  "body": "残り{{item.quantity}}点 — 発注点は{{item.reorder_level}}でした",
  "notificationType": "time-sensitive"
}
```

### 5. 外部サービスによる監視とアップタイム通知

**トリガー**：Datadog、Better Uptime、PagerDuty、Freshping — 監視サービスからWebhookを受け取り、電話呼び出しにエスカレーションしたい場合

監視ツールがMakeにWebhookを送る場合、アラートを強化できます——追加のコンテキストを取得したり、ステータスページを確認したり、データベースに記録したりして、フル情報のEchobell通知を送ることができます。

```json
{
  "title": "アラート: {{monitor.name}} がダウンしています",
  "body": "エラー: {{monitor.error}} — ダウン開始 {{monitor.started_at}}",
  "notificationType": "calling"
}
```

## 適切な通知タイプを選ぶ

3つの選択肢があります。正しく選べばアラートが役立ち続け、うるさくなりません：

| タイプ | 動作 | 最適な用途 |
|---|---|---|
| `active` | 標準プッシュ通知 | 定常的な情報更新 |
| `time-sensitive` | iOS集中モードを突き抜ける | 数分以内に対応すべき重要事項 |
| `calling` | 着信のように電話が鳴る | 売上や本番環境に影響する障害 |

`calling`タイプにはプレミアムサブスクリプションが必要です——火災報知器と同じ感覚で扱ってください。すべてが鳴り続けると、何も意味を持たなくなります。

## チームでアラートを共有する

通知を複数人に届ける必要がある場合、Echobellのチャンネルモデルがそのまま使えます。チャンネルを作成し、受け取ってほしい人たちにサブスクリプションリンクを共有します。シナリオが発火すると、サブスクライブしている全員にアラートが届きます。

複雑なルーティングルールを設定することなく、小規模チームの共有監視を運用できるシンプルな方法です。オンコールの担当者が変わったら、旧担当者はサブスクライブ解除して、新担当者がサブスクライブするだけ。以上です。

---

## もうひとつ：Make + メールトリガー

すべてのシステムがMakeと連携できるわけではありません。メールしか送れないシステムもあります。Echobellは各チャンネルに独自の`@echobell.app`メールアドレスを付与しています。Makeでいれ Gmailの受信トレイを監視し、特定の送信者や件名でフィルタリングして、一致するメールをEchobellに転送することもできます。あるいは、Makeを介さずにチャンネルアドレスに直接転送することも可能です。

どちらの方法でも、メールはEchobell通知になり、プッシュまたは電話呼び出しになります。
