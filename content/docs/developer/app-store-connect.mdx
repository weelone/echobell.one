# App Store Connect Webhook Integration

App Store Connect webhooks broadcast every meaningful change in your iOS release pipeline. When those signals flow into Echobell, your team sees review decisions, releases, and TestFlight feedback the moment they happen‚Äîno manual refreshes or inbox digging required.

**Why teams wire this integration:**

- Faster reaction time when reviews flip to "Rejected" or "Ready for Sale"
- A single notification stream for TestFlight feedback and build processing
- Clear, mobile-friendly alerts that keep PMs, QA, and marketing aligned

<Callout type="info">
  App Store Connect webhooks are available to developers with Account Holder,
  Admin, or App Manager roles. Each webhook can only apply to one app, and you
  can create up to ten webhooks per app.
</Callout>

## Prerequisites

- Echobell channel with webhook triggers enabled
- App Store Connect access (Account Holder, Admin, or App Manager)
- At least one active iOS app to monitor
- Your Echobell webhook URL (Channel ‚Üí **Triggers**)

## What You Can Monitor

App Store Connect can ping Echobell for:

1. **App Store version lifecycle** ‚Äî submission progress, approvals, rejections, releases
2. **TestFlight milestones** ‚Äî build processing results, tester feedback, invite state changes

## Setup Overview

1. Prep an Echobell channel plus notification templates
2. Copy the channel webhook URL
3. Configure a webhook in App Store Connect
4. Send a test delivery to verify formatting

### 1. Create an Echobell Channel

1. Open the Echobell app
2. Create a new channel (e.g., "App Store Connect")
3. Pick a distinctive color for quick scanning
4. Add teammates who should receive these updates

### 3. Get Your Echobell Webhook URL

1. Open your channel settings ‚Üí **Triggers**
2. Copy the webhook URL provided
3. Keep it private; you'll paste it into App Store Connect

### 4. Configure App Store Connect Webhook

Follow [Apple's official guide](https://developer.apple.com/help/app-store-connect/manage-your-team/manage-webhooks/) with the Echobell URL you copied earlier:

#### Create a Webhook

1. Sign in to [App Store Connect](https://appstoreconnect.apple.com/)
2. Navigate to **Users and Access** ‚Üí **Integrations**
3. Under **Additional**, click **Webhooks**
4. Click the add button (+) next to Webhooks
5. Configure your webhook:
   - **Name**: Descriptive label (e.g., "Echobell Notifications")
   - **Payload URL**: Paste your Echobell webhook URL
   - **Secret**: Create a secure string for verification
   - **App**: Select the app you want to monitor
   - **Event Triggers**: Pick the events you want to follow

#### Available Event Triggers

Mix and match what matters:

**App Store Version Events**

- **App Store Version State Changed** ‚Äî every movement through review states
- **App Store Version Released** ‚Äî final push to the store

**TestFlight Events**

- **TestFlight Beta Feedback** ‚Äî new comments, ratings, and device details
- **TestFlight Beta Review State Changed** ‚Äî build approvals and rejections

7. Click **Add** to create the webhook

### 5. Test Your Integration

Use App Store Connect's built-in **Test** button:

1. In the webhook list, open the webhook you just added
2. Click **Test** ‚Üí App Store Connect sends a ping
3. Confirm Echobell shows the "Ping" event with your template formatting

## Advanced Configuration

### Webhook Security

1. **HTTPS**: Echobell URLs ship with TLS; no extra setup required.
2. **Secret handling**: App Store Connect enforces a secret, but Echobell treats it as metadata. Store it so you can validate signatures later if you prefer.
3. **IP allowlisting**: Tighten firewalls by allowing Apple's outbound ranges only.

## Managing Webhooks

### View Webhook Deliveries

1. Go to **Users and Access** ‚Üí **Integrations** ‚Üí **Webhooks**
2. Click your webhook
3. Review **Recent Deliveries** for event IDs, statuses, timestamps, and responses

### Webhook Delivery Status

| Status  | Description                                    |
| ------- | ---------------------------------------------- |
| Success | Webhook reached Echobell and returned 2xx      |
| Pending | Apple is still delivering the event            |
| Failed  | Delivery error (resend after resolving issues) |

### Resend Failed Deliveries

1. In **Recent Deliveries**, locate the failed event
2. Click **Resend**
3. Confirm the new delivery shows Success

### Modify Webhook Settings

- Open the webhook, adjust the name, secret, payload URL, or triggers
- Click **Save** ‚Äî Apple applies the change right away

### Temporarily Disable Webhooks

Need downtime? Use **Turn Off Webhook** and bring it back later with **Turn On Webhook**.

## Use Cases and Notification Strategies

**App release readiness** ‚Äî Get notified when builds shift from *In Review* to *Ready for Sale* so you can publish release notes or run marketing plays.

**TestFlight triage** ‚Äî Pipe every tester comment into a dedicated channel and loop in QA automatically.

**Portfolio oversight** ‚Äî Large orgs typically split notifications by product and environment, plus an escalation-only channel for blockers.

Tweak your approach:

- **Small teams:** One shared channel, verbose templates, everyone subscribed.
- **Larger orgs:** Channels per platform or market, role-based membership, summarized digests for stakeholders.

## Troubleshooting

### Common Issues

**Not receiving notifications**

1. Verify the Echobell webhook URL in App Store Connect
2. Confirm relevant event triggers are selected
3. Review **Recent Deliveries** for errors
4. Make sure the Echobell channel is active and members are subscribed

**Webhook authentication errors**

1. Double-check the shared secret
2. Ensure your webhook endpoint is publicly accessible and SSL-valid
3. Inspect payload signing logic if you added custom validation

**Missing event data**

1. Confirm template variable names
2. Check that the event type actually includes the data you expect
3. Log the raw payload to debug parsing issues

### Testing Webhooks

- **App Store Connect test ping** ‚Äî fastest sanity check.
- **Real event** ‚Äî submit a TestFlight build or app update and watch deliveries.

### Rate Limiting and Performance

- Apple may throttle if a channel generates heavy traffic.
- Echobell queues inbound events to avoid dropped payloads.
- Spread events across channels if you manage dozens of apps.

## Best Practices

**Security**

- Keep Echobell URLs private and rotate secrets every few quarters.
- Audit webhook delivery logs for unusual spikes.

**Organization**

- Use naming patterns like `Product - Environment` so channels self-describe.
- Document webhook settings in your runbook for quick onboarding.

**Monitoring**

- Schedule quarterly test pings.
- Enable alerts for repeated failures and review volume trends to avoid alert fatigue.
