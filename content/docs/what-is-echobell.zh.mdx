# 什么是 Echobell？

您是否曾经错过重要的生产环境警报，因为它被埋在其他 47 个通知下面？或者凌晨 2 点被"磁盘空间不足"的警告吵醒，但这个警告其实可以等到早上？我们也遇到过这种情况，正是这种挫败感促使我们创建了 Echobell。

**Echobell 是一款即时警报应用，终于让开发者通知为您服务，而不是与您作对。** 它将来自您系统的混乱的 webhook 和电子邮件流转换为清晰、可操作的警报，在正确的时间以正确的紧急程度到达您那里。

## 我们解决的问题

作为开发者和运维工程师，我们生活在一个充满不断响起的提示音、警报和通知的世界中。Slack 消息、转发邮件、PagerDuty 警报、Grafana 通知、GitHub webhook —— 永无止境。结果如何？**通知疲劳**。我们开始忽略一切，这意味着我们会错过真正重要的警报。

传统的通知系统对待所有事物都一视同仁。成功的部署获得与数据库宕机相同的关注度。代码审查请求看起来与安全漏洞警报完全相同。

**Echobell 通过为您提供对如何以及何时被打断的精细控制来改变这种状况。**

## Echobell 的特别之处

### 智能通知管理
- **即时 webhook 通知**，将任何 HTTP 调用转换为格式完美的移动警报
- **邮件到通知的桥梁**，将重要消息从邮件地狱中拯救出来
- **自定义警报频道**，按项目、严重程度或团队组织
- **智能紧急级别** —— 从静默更新到"立即叫醒我"的紧急呼叫

### 开发者优先设计
- **基于模板的通知**，使用来自您的 webhook 载荷的变量
- **零配置集成**，与 GitHub Actions、Grafana 和自定义脚本等工具集成
- **团队协作功能**，用于共享通知频道
- **隐私优先架构** —— 您的警报内容永远不会到达我们的服务器

### 解决现实世界的问题
Echobell 不是又一个需要检查的仪表板，而是直接将重要信息传递到您的手机、智能手表或您需要的任何地方。这就是从愤怒客户邮件中发现故障与在故障发生的那一秒就捕获它之间的区别。

## Echobell 实际是如何工作的

这就是神奇之处：您的系统（监控工具、CI/CD 管道、自定义脚本）向 Echobell 发送 webhook 请求或电子邮件。我们处理该数据并立即向订阅该频道的每个人提供格式化的通知。

**流程很简单：**
1. 您的系统中发生了某些事情（部署、错误、阈值突破）
2. 您的系统调用 Echobell webhook 或发送电子邮件
3. Echobell 处理数据并应用您的通知模板
4. 格式化的通知立即传递给所有订阅者
5. 您可以立即采取行动，并获得所需的所有上下文

**是什么让它变得强大：** 通知内容是从您的系统发送的实际数据动态生成的，所以您得到的不是通用警报，而是可操作的信息，如服务器名称、错误代码、仪表板链接和特定指标。

## 构建块：理解 Echobell 的核心概念

### 频道：您的通知管道

频道是您的个性化通知管道。可以将其视为专门针对特定类型警报的电台。每个频道：

- **有清晰的视觉识别**，使用名称和颜色进行即时识别
- **包含智能模板**，将原始数据转换为有用的通知
- **可以通过多种方式触发** —— webhook 用于自动化，电子邮件用于灵活性
- **通过安全订阅链接轻松分享**
- **尊重隐私** —— 只有元数据存在我们的服务器上

### 通知类型：在正确的时间发送正确的消息

订阅频道时，您选择希望如何被中断：

- **正常（活跃）**：尊重您的专注设置的标准应用通知
- **时效性**：可以突破某些专注模式的高优先级通知
- **呼叫**：适合需要立即关注的关键情况的类似来电的警报

**现实世界示例：** 部署更新使用"正常"，数据库故障使用"时效性"，安全事件使用"呼叫"。

### 模板变量：动态数据，不是静态文本

使用模板中的变量使您的通知变得动态：

```
标题：🚨 {{server_name}} 遇到问题！
内容：CPU {{cpu_usage}}% | 内存：{{memory_usage}}% | 检查：{{dashboard_link}}
```

当使用正确的变量触发时，这些模板生成个性化通知，如：
"🚨 Production-Web-01 遇到问题！CPU 94% | 内存：87% | 检查：https://grafana.company.com/d/server-health"

### 触发方法：为您的基础设施提供灵活性

#### Webhook 触发：纯自动化

集成任何可以进行 HTTP 调用的系统。通过查询字符串或 JSON 主体传递变量：

```http
GET https://hook.echobell.one/t/xxx?server_name=Production&cpu_usage=95&dashboard_link=https://grafana.company.com/d/server-health

# 或者

POST https://hook.echobell.one/t/xxx
Content-Type: application/json

{
  "server_name": "Production", 
  "cpu_usage": 95,
  "error_message": "检测到高内存使用率",
  "dashboard_link": "https://grafana.company.com/d/server-health"
}
```

**可用的特殊变量：**
- `externalLink`：在通知记录中创建可点击的链接
- `bodyAsText`：用于纯文本请求内容
- `header`：访问 HTTP 头信息，如 `{{header["user-agent"]}}`

#### 电子邮件触发：当自动化不可能时

有时您无法修改源系统。电子邮件触发拯救了这一天 —— 只需向您频道的地址发送电子邮件。Echobell 自动提取：

- `from`：发件人电子邮件地址
- `subject`：电子邮件主题行
- `text`：纯文本内容
- `html`：富格式的 HTML 内容

**现实世界示例：** 某些遗留系统只支持电子邮件通知。与其不断检查您的邮箱，不如将它们转发到 Echobell 频道并接收清洁的移动通知。

## 隐私和数据存储：您的秘密保持为您的秘密

Echobell 采用**隐私优先**的方法设计：

**存在我们服务器上的内容：**
- 用户数据和身份验证
- 频道配置（名称、颜色、模板）
- 订阅信息

**保留在您设备上的内容：**
- 实际的通知内容
- 警报历史和记录
- 来自您系统的敏感数据

这种方法意味着即使我们的服务器被攻破，您的敏感生产数据也保持在您的设备上安全。
